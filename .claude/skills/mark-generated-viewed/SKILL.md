# mark-generated-viewed

GitHub PR の自動生成ファイルを検出し、Files changed タブで viewed にマークする leaf skill ([rules/skills.md](../../../rules/skills.md))。手書きのロジック・テストだけがレビュー対象として残るため、生成物の差分を毎回手でたたんでいく手間がなくなる。

検出は `scripts/detect-generated.ts`（読み取り専用）が行い、マークは `markFileAsViewed` の for ループを発行する。

## 前提条件

- `gh` が認証済みであること（PR の読み取り・マークに使う）

## ワークフロー

```text
- [ ] Step 1: detect-generated.ts で生成物を検出
- [ ] Step 2: 検出結果をユーザーに提示
- [ ] Step 3: markFileAsViewed の for ループで viewed にマーク
- [ ] Step 4: マーク件数とレビュー対象を報告
```

### Step 1: 検出

検出結果は後続 Step で shell コードに直接埋め込まず JSON ファイル経由で扱うため、リダイレクトで保存する:

```bash
bun --config=/dev/null .claude/skills/mark-generated-viewed/scripts/detect-generated.ts <pr-number-or-url> [--repo owner/repo] > .claude/tmp/mark-generated-viewed-result.json
```

- `<pr-number-or-url>`: PR 番号（例: `768`）または PR URL（例: `https://github.com/owner/repo/pull/768`）
- 番号のみ指定時は `--repo owner/repo`、未指定ならカレントディレクトリの repo を使う
- URL に `#`（コメントリンクの fragment）や `?` を含む場合は `"..."` で囲む（未クォートだと shell が `#` 以降をコメント扱いする）。例: `... detect-generated.ts "https://github.com/owner/repo/pull/768#issuecomment-1"`

出力 JSON の主要フィールド:

```json
{
  "repo": "owner/repo",
  "pr_number": 768,
  "pr_node_id": "PR_kwDO...",
  "head_sha": "...",
  "total_changed": 85,
  "generated": [ { "path": "api/docs/docs.go", "reason": "content:go-codegen" } ],
  "review": [ "api/usecase/interactor/article.go" ]
}
```

### Step 2: 提示

`.claude/tmp/mark-generated-viewed-result.json` を Read し、`generated`（マーク対象）と `review`（手書き = レビュー対象に残す）を要約してユーザーに示す。`reason` 別の内訳（`content:*` / `name:*` / `gitattributes:*`）を添えると、特に `name:*`（内容未確認の名前判定）の妥当性をユーザーが確認できる。

### Step 3: マーク

`generated[].path` を **JSON ファイルから `jq` で NUL 区切りの一覧として書き出し、`while read -d ''` で 1 件ずつ読んでループする**（パス文字列を shell コードのリテラルとして直接埋め込むと、ファイル名に `$()` / backtick / `"` / `\` を含む場合にコマンド置換や shell 構文破壊が起きうる。`mapfile` は bash 専用ビルトインで zsh には無いため使わない）。GraphQL 側もパスは変数として渡す（文字列リテラルに直接埋め込むと `"` や `\` を含むパスでクエリが壊れる）:

```bash
jq --raw-output0 '.generated[].path' .claude/tmp/mark-generated-viewed-result.json > .claude/tmp/mark-generated-viewed-paths.bin

while IFS= read -r -d '' p; do
  gh api graphql \
    -F pr='<PR_NODE_ID>' \
    -F path="$p" \
    -f query='
      mutation($pr: ID!, $path: String!) {
        markFileAsViewed(input: {pullRequestId: $pr, path: $path}) { pullRequest { id } }
      }' | head -1
done < .claude/tmp/mark-generated-viewed-paths.bin
```

- `<PR_NODE_ID>` を `pr_node_id` の値に置換する
- `generated` が空なら何もマークしない（`.claude/tmp/mark-generated-viewed-paths.bin` が空になり loop は 0 回）
- **`| head -1` は出力を 1 行に絞るだけで、パイプライン全体の exit code は `head` のもの（常に 0）になる**。`gh api graphql` 自体が認証切れ・権限不足・GitHub が拒否するパス等で失敗しても exit code では検知できないため、各行の出力に `"errors"` キーが含まれていないかを確認して成功を判定すること（`"errors"` があれば該当パスは実際には viewed マークされていない）

### Step 4: 報告

マークした件数（`reason` 別内訳）と、レビュー対象として残った `review` の件数を報告する。

## 検出シグナル

3 つのシグナルで判定する（`scripts/detect-generated.ts` が import する [`../_shared/generated-detect.ts`](../_shared/generated-detect.ts) 参照）:

| シグナル | reason prefix | 例 |
|----------|---------------|-----|
| ファイル先頭マーカー | `content:` | `// Code generated ... DO NOT EDIT` / `@generated` / `Generated by orval` / auto-generated 文言 |
| ファイル名/パス規則 | `name:` | lock files / `*.pb.go` / `*_pb2.py` / `swagger.{json,yaml}` / `*.gen.go` |
| root `.gitattributes` | `gitattributes:` | `linguist-generated` 指定（GitHub が collapse 判定に使う正準シグナル） |

判定優先順位: **`.gitattributes` の明示指定（`linguist-generated` / `-linguist-generated`）が最優先**。該当ルールがあれば名前/内容判定より先に確定する（`-linguist-generated` の明示 opt-out はロックファイル等の名前パターンにマッチしても `review` 側に残る）。`.gitattributes` に該当ルールが無いファイルのみ、ファイル名 → 内容マーカーの順で判定する。

注意点:

- **`name:*` は内容を読まずファイル名だけで判定する**ため、手書きの `openapi.yaml` 等を誤検出しうる。Step 2 でユーザーに提示し妥当性を確認する
- viewed マークは GitHub UI でいつでも解除できる（低リスク・可逆）
- 内容マーカーはファイル先頭 40 行のみ走査する（生成物のヘッダは常に先頭に出るため）
- nested `.gitattributes` は未対応（root のみ）。removed ファイルは内容判定できないため名前規則のみ適用する
- `.gitattributes` の glob は common subset のみ対応。スペースを含むパターン（`my\ dir/*` 等の escape / quote）は未対応で、該当する生成ファイルは検出から漏れて `review` 側に残りうる

## トラブルシューティング

| 問題 | 対処 |
|------|------|
| `PR ... を取得できません` (exit 3) | PR 番号・`--repo` を確認。番号のみ指定時はカレント repo が対象 |
| `gh 認証エラー` (exit 2) | `gh auth login` で認証する |
| 手書きファイルが `generated` に入る | `name:*` の誤検出。当該ファイルは Step 3 のループから外す。再発するなら検出規則の見直しを検討 |
| 生成物が `review` に残る | 先頭 40 行にマーカーが無く名前規則にも該当しない。`reason` が付く形にマーカー/規則の追加を検討 |
